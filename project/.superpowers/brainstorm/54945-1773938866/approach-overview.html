Approach: How should we build this?

Three strategies for implementing the web GTD app on top of your existing Vue + shadcn-vue scaffolding

A

Recommended: Screen-by-Screen, API-Connected

Build each page fully functional, one at a time. Each page gets its own Pinia store, API service, and components. Start with Inbox (core GTD loop), then expand outward.

Pros

  • Usable after first page ships
  • Each page is independently testable
  • Matches your backend's REST structure

Cons

  • Shared patterns may need refactoring as they emerge
B

Foundation First

Build all shared infrastructure upfront — API client, auth, stores, reusable task components, sidebar — then fill in each page. More planning, later payoff.

Pros

  • Consistent from day one
  • Shared components designed holistically

Cons

  • Nothing works until a lot is built
  • Risk of over-engineering shared layers
C

Static UI First, Then Wire Up

Build all pages with hardcoded mock data (like the HTML prototype), then connect APIs in a second pass. Visual progress fast, functionality later.

Pros

  • Fast visual validation
  • Design issues caught early

Cons

  • Rework when connecting real data
  • Two passes over every page
Proposed Build Order (Approach A)
Phase 1 — Core Shell
  1. Auth (login, signup, OTP, JWT handling)
  2. App shell (sidebar nav, routing, layout)
  3. API client + auth interceptor
Phase 2 — GTD Core
  1. Inbox (capture + triage)
  2. Next Actions (context groups + filters)
  3. Task detail side panel
Phase 3 — Expand
  1. Projects (list + detail)
  2. Waiting For
  3. Someday/Maybe
  4. Weekly Review wizard
